# SmartToDo1 - Техническое описание проекта

## 1. Паспорт проекта

**Название:** SmartToDo1  
**Тип системы:** веб-приложение (клиент-серверная архитектура)  
**Предметная область:** управление персональными задачами с авторизацией и ролью администратора  
**Модель доступа:** role-based access control (RBAC)  
**Подход к разработке:** разделение на frontend и backend с четкими API-контрактами

### 1.1. Цель проекта
Создать удобную и устойчивую систему для:
- создания и сопровождения списка задач;
- фильтрации/сортировки и постраничного просмотра задач;
- подтверждения email при регистрации;
- управления пользователями через административный интерфейс;
- предоставления локализованного интерфейса (RU/EN).

### 1.2. Решаемые задачи
Проект решает следующие практические задачи:
- безопасная регистрация и вход пользователей;
- хранение и обновление задач в БД;
- защита API через JWT-токены;
- разграничение прав `User` и `Admin`;
- предотвращение повторных запросов при кликах в UI;
- централизованный мониторинг через логи backend.

## 3. Архитектура решения

## 3.1. Общая схема
Система построена по классической схеме:
1. Frontend отправляет HTTP-запросы к backend.
2. Backend валидирует токены и права доступа.
3. Сервисный слой обрабатывает бизнес-логику.
4. Репозитории обращаются к БД через EF Core.
5. Результат возвращается на frontend и отображается в UI.

## 3.2. Архитектура frontend
Ключевые модули:
- `components/` — визуальные компоненты (модалки, список задач, хедер, фильтры, пагинация).
- `hooks/` — логика сценариев (`useAuth`, `useTaskManager`, `useTaskStorage`).
- `services/` — API-слой и обертки запросов (`api.ts`, `authApi.ts`, `taskApi.ts`).
- `i18n/` — словарь переводов и языковые ключи.
- `styles/` — стилевые пакеты по компонентам.

Главный orchestration-layer на клиенте: `useTaskManager`.
Этот хук координирует:
- загрузку задач,
- фильтрацию/сортировку/пагинацию,
- открытие/закрытие модалок,
- состояние темы,
- состояние языка интерфейса,
- доступ к админ-модалке пользователей.

## 3.3. Архитектура backend
Backend разделен на слои:
- `API/Controllers` — HTTP endpoints.
- `Application/Services` — бизнес-логика.
- `Infrastructure/Repositories` — доступ к данным.
- `Infrastructure/Databases` — контекст БД и сидирование.
- `Domain/Models` — доменные сущности.

Это снижает связанность кода и упрощает поддержку.

## 4. Модель данных

## 4.1. Пользователь (`Users`)
Основные поля:
- `Id` (Guid)
- `Login` (string)
- `PasswordHash` (string)
- `Email` (string)
- `UserRole` (enum `User` / `Admin`)
- `EmailConfirmed` (bool)
- `VerificationCode` (nullable string)
- `VerificationCodeExpiry` (nullable DateTime)

## 4.2. Задача (`TaskModel`)
Основные поля:
- `Id`
- `Title`
- `Description`
- `Priority`
- `IsCompleted`
- `CreatedAt`
- `UserId`

Связь:
- один пользователь -> много задач (`Users` 1:N `TaskModel`).

## 4.3. Refresh token (`TokenModel`)
Содержит:
- хеш refresh-токена,
- владельца (`UserId`),
- срок действия,
- признак отзыва.

Используется для обновления access-токена без повторного логина.

## 5. API-контракты

## 5.1. Пользователи
- `POST /api/users/Registration` — регистрация.
- `POST /api/users/Auth` — авторизация.
- `POST /api/users/AuthToken` — подтверждение email кодом.
- `POST /api/users/CodeGeneration?email=` — повторная отправка кода.
- `POST /api/users` — refresh токена.
- `GET /api/users/sercer` — проверка access token.
- `GET /api/users` — список пользователей (только `Admin`).

## 5.2. Задачи
- `GET /api/tasks` — получение задач с query-параметрами.
- `POST /api/tasks/Addendum` — добавление задачи.
- `PATCH /api/tasks/Update/{id}` — редактирование.
- `DELETE /api/tasks/Delete/{id}` — удаление.

---

## 6. Реализованная бизнес-логика

## 6.1. Регистрация и подтверждение почты
Текущий процесс:
1. Пользователь отправляет форму регистрации.
2. Backend создает запись пользователя.
3. Backend генерирует код и отправляет email.
4. Frontend переводит пользователя в окно верификации.
5. Пользователь вводит только код (email недоступен для редактирования).

Важно: лишняя повторная отправка кода после регистрации на frontend удалена.

## 6.2. Авторизация и refresh
- после логина frontend получает `accessToken` и `refreshToken`;
- токены хранятся в cookies;
- при невалидном access вызывается refresh;
- для `403` (email не подтвержден) UI переключается в режим верификации.

## 6.3. Работа с задачами
- CRUD операции с серверной синхронизацией;
- устойчивость к race-condition при удалении/обновлении;
- кнопки блокируются во время запросов, чтобы исключить дубль-клики.

## 6.4. Админ-панель пользователей
- доступ только для роли `Admin`;
- отображение роли нормализовано:
  - `1`/`Admin` -> `Admin`
  - `0`/`User` -> `User`
- отображение валидности email.

## 9. Безопасность

Реализованные меры:
- пароли в БД хранятся только как BCrypt hash;
- JWT для защищенных endpoint;
- роль `Admin` проверяется сервером через `[Authorize(Roles = "Admin")]`;
- refresh token хранится в БД в хешированном виде;
- централизованная обработка ошибок middleware.

Ограничения текущей dev-конфигурации:
- CORS ориентирован на локальный origin;
- часть endpoint и сообщений требует дальнейшего hardening для production.

## 10. Логирование и диагностика

Backend логирует:
- начало/завершение запросов;

## 11. Тестовые пользователи (seed)

В проекте настроено автоматическое создание двух тестовых пользователей:
- `admin / 123456` (роль: `Admin`, email: `admin@smarttodo.local`)
- `user / 123456` (роль: `User`, email: `user@smarttodo.local`)

Параметры seed-пользователей:
- `EmailConfirmed = true`;
- пароль сохраняется как BCrypt hash;
- создаются только если отсутствуют в БД.

### 11.1. Где находится логика seed
- `backend/RAA/Infrastructure/Databases/DbSeeder.cs`

### 11.2. Где вызывается seed
- `backend/RAA/Program.cs`
- вызов: `await DbSeeder.SeedAsync(app);`

Таким образом, логика создания `admin/user` **вынесена в отдельный файл**, а не захардкожена в `Program.cs`.
